# PatentClaw 项目说明书

> **PatentClaw（专利龙虾）** — 基于多 Agent 编排的中国发明专利申请文件自动生成系统

---

## 一、项目概述

### 1.1 项目定位

PatentClaw 是一个运行在 [OpenClaw](https://github.com/nicepkg/openclaw) 平台上的 AI Skill，专门面向**中国发明专利申请**场景。它解决科研人员在专利撰写中面临的核心痛点：

- 从学术论文到专利文件的**思维模式转换**（论文追求公开复现，专利追求抽象保护）
- 权利要求书与说明书之间的**术语一致性**维护
- 权利要求**保护范围策略**的选择与权衡
- 符合国家知识产权局（CNIPA）排版规范的**格式化输出**

### 1.2 核心能力

| 能力 | 说明 |
|------|------|
| 论文解析与上位化 | 自动从学术论文中提取技术方案，抽象为专利语言 |
| 多策略权利要求生成 | 并行生成宽保护、稳妥、窄保护 3 套策略供用户选择 |
| 现有技术检索 | 自动搜索相关专利和论文，评估新颖性风险 |
| 术语一致性审校 | 6 类检查规则，自动修复术语不一致 |
| 国知局格式导出 | 输出符合 CNIPA 排版要求的 .docx 文件 |

### 1.3 适用场景

- 高校科研团队将论文成果转化为专利申请
- 企业研发部门批量撰写技术交底书
- 专利代理机构辅助初稿生成
- 个人发明人快速生成专利申请文件初稿

## 二、系统架构

### 2.1 整体架构

PatentClaw 采用**编排器 + 4 个专业子 Agent** 的多 Agent 协作架构：

```
┌───────────────────────────────────────────────────────────┐
│                    PatentClaw 编排器                        │
│                     (SKILL.md)                             │
│                                                           │
│   ┌─────────┐   ┌──────────┐   ┌──────────┐              │
│   │ Phase 1  │──→│ Phase 3  │──→│ Phase 5  │              │
│   │ 交互分析 │   │ 用户决策 │   │ 导出交付 │              │
│   └─────────┘   └──────────┘   └──────────┘              │
│        │              ▲              ▲                     │
│        ▼              │              │                     │
│   ┌─────────┐   ┌──────────┐                              │
│   │ Phase 2  │   │ Phase 4  │                              │
│   │ 并行生成 │   │ 自动审校 │                              │
│   └─────────┘   └──────────┘                              │
└───────┬───────────────┬──────────────────────────────────┘
        │               │
        ▼               ▼
┌───────────────┐ ┌───────────────┐
│  sessions_spawn │ │  sessions_spawn │
└───┬───┬───┬───┘ └───────┬───────┘
    │   │   │             │
    ▼   ▼   ▼             ▼
  ┌───┐┌───┐┌───┐    ┌────────┐
  │ A ││ B ││ C │    │   D    │
  └───┘└───┘└───┘    └────────┘

A = 权利要求策略师    C = 现有技术检索员
B = 说明书撰写师      D = 审校修复师
```

### 2.2 技术栈

| 层级 | 技术 |
|------|------|
| AI 平台 | OpenClaw Agent Platform |
| 子 Agent 调度 | `sessions_spawn` 原生工具 |
| 一致性检查 | `check-consistency.js`（Node.js，正则 + 相似度算法） |
| 文档导出 | `export-docx.js`（Node.js + [docx](https://www.npmjs.com/package/docx) 库） |
| 网络搜索 | `web_search` 工具（现有技术检索员使用） |

### 2.3 子 Agent 通信协议

各子 Agent 通过在输出中嵌入带标记的 JSON 数据与编排器通信：

| 子 Agent | 输出标记 |
|----------|---------|
| 权利要求策略师 | `__CLAIMS_RESULT__{...}__CLAIMS_RESULT__` |
| 说明书撰写师 | `__SPEC_RESULT__{...}__SPEC_RESULT__` |
| 现有技术检索员 | `__PRIOR_ART_RESULT__{...}__PRIOR_ART_RESULT__` |
| 审校修复师 | `__FIXED_RESULT__{...}__FIXED_RESULT__` |

编排器从子 Agent 的输出文本中解析标记，提取结构化 JSON 数据用于后续流程。

## 三、工作流程

### 3.1 模式一：论文 → 技术交底书

此模式由编排器直接处理，无需子 Agent 参与。

```
用户提供论文
    │
    ▼
┌──────────────┐
│  第一步：论文解析  │  提取标题、作者、技术背景、方案、实验结果、附图
└──────┬───────┘
       ▼
┌──────────────┐
│  第二步：创新点识别 │  提炼 2-5 个核心创新点
└──────┬───────┘
       ▼
┌──────────────┐
│  第三步：上位化改写 │  具体描述 → 抽象概念（宽保护版 + 稳妥版）
└──────┬───────┘
       ▼
┌──────────────┐
│  第四步：生成交底书 │  输出结构化技术交底书 + 导出 .docx
└──────────────┘
```

**上位化改写示例：**

| 论文原文 | 上位化表述 |
|---------|-----------|
| 使用 220nm SOI 平台 | 基于绝缘体上半导体平台 |
| 微环谐振器 | 基于谐振结构的波长选择单元 |
| 采用 ResNet-50 网络 | 采用深度卷积神经网络 |
| 使用 COMSOL 进行仿真 | 通过数值仿真验证 |

### 3.2 模式二：交底书 → 专利四件套

此模式采用多 Agent 并行编排，分 5 个阶段执行。

#### Phase 1：前置准备

编排器与用户交互，收集必要信息：

```
必须由用户提供（绝不编造）：
├── 发明人姓名及排序
├── 申请人名称（正式全称）
├── 联系地址和邮编
├── 联系电话和邮箱
└── 代理机构和代理人（可标注"待定"）
```

编排器将交底书内容和申请人信息组装为 `disclosure_context` JSON，作为子 Agent 的输入。

#### Phase 2：并行生成

编排器同时 spawn 3 个子 Agent，并行工作：

```
                    disclosure_context
                    ┌──────┼──────┐
                    │      │      │
                    ▼      ▼      ▼
            ┌──────────┐┌──────┐┌──────────┐
            │ Agent A  ││Agent B││ Agent C  │
            │          ││      ││          │
            │ 3套权利  ││说明书││ 查新报告 │
            │ 要求策略  ││+ 摘要││ + 建议   │
            └──────────┘└──────┘└──────────┘
                 │          │        │
                 └──────────┼────────┘
                            ▼
                     等待全部返回
```

**并行优势**：3 个 Agent 各自拥有独立的上下文窗口，不互相干扰。权利要求的发散式策略探索、说明书的收敛式细节展开、网络搜索的 I/O 等待，三者同时进行。

#### Phase 3：用户决策

编排器汇总 3 个子 Agent 的结果，向用户展示：

1. **查新报告**：相关现有技术、相似度评估、新颖性风险
2. **3 套策略对比**：

| 维度 | 宽保护 | 稳妥保护 | 窄保护 |
|------|--------|---------|--------|
| 独立权利要求特征数 | 1-2 个 | 3-4 个 | 5-6 个 |
| 从属权利要求数 | 3-4 条 | 4-6 条 | 2-3 条 |
| 保护范围 | 最大 | 适中 | 较小 |
| 授权可能性 | 较低 | 较高 | 最高 |

3. **综合建议**：结合查新结果推荐策略

用户选择后，编排器组装完整的四件套 JSON。

#### Phase 4：自动审校

编排器 spawn 审校修复师，自动完成：

```
输入：四件套 JSON
    │
    ▼
┌──────────────────┐
│ 运行 check-consistency.js │
└──────┬───────────┘
       ▼
┌──────────────────┐
│ 修复 error 级别问题  │  术语统一、编号修正、摘要精简
└──────┬───────────┘
       ▼
┌──────────────────┐
│ 修复 warning 级别问题│  补充说明书覆盖、优化权利要求
└──────┬───────────┘
       ▼
┌──────────────────┐
│ 重新检查（最多 3 轮）│  直到 error = 0
└──────┬───────────┘
       ▼
输出：修正后的四件套 + 修复报告
```

#### Phase 5：导出交付

编排器完成最终输出：

1. 展示修复报告摘要
2. 展示最终四件套 Markdown
3. 调用 `export-docx.js` 导出 .docx 文件
4. 提醒用户提交前须经专利代理师审核

## 四、一致性检查引擎

### 4.1 检查规则

| 编号 | 检查类型 | 级别 | 说明 |
|------|---------|------|------|
| C1 | `term_inconsistency` | error | 权利要求书与说明书使用不同术语描述同一概念 |
| C2 | `claim_overlap` | error | 从属权利要求的附加特征与独立权利要求重复 |
| C3 | `claim_ref_forward` | error | 从属权利要求引用了编号更大的权利要求 |
| C4 | `claim_ref_invalid` | error | 引用了不存在的权利要求编号 |
| C5 | `abstract_too_long` | error | 说明书摘要超过 300 字 |
| W1 | `term_missing_in_spec` | warning | 权利要求中的术语在说明书中未出现 |
| W2 | `spec_insufficient_coverage` | warning | 说明书对权利要求特征覆盖不足 |
| W3 | `claim_near_overlap` | warning | 从属权利要求与独立权利要求高度相似 |
| W4 | `independent_claim_narrow` | warning | 独立权利要求限定特征过多 |
| W5 | `independent_claim_too_specific` | warning | 独立权利要求含具体数值参数 |
| W6 | `dependent_claims_overlap` | warning | 两条从属权利要求附加特征相似 |
| I1 | `abstract_no_drawing` | info | 摘要未指定摘要附图 |

### 4.2 智能过滤

内置专利法律套话停用词表，避免将结构性短语误判为术语不一致：

```
停用词示例：其特征在于、根据权利要求、包括以下步骤、
任一项所述、本发明涉及、本领域技术人员...
```

### 4.3 相似度算法

使用基于公共子串的相似度计算，配合字符重叠率判断，识别近义替换：

- 「全局池化」vs「全局平均池化」→ 相似度 0.75 → 标记为术语不一致
- 「卷积神经网络」vs「深度学习模型」→ 相似度 0.2 → 不标记

## 五、文档导出规范

### 5.1 CNIPA 排版要求

| 要素 | 规范 |
|------|------|
| 纸张 | A4（210mm x 297mm） |
| 页边距 | 上下 25.4mm，左右 25mm |
| 文档标题 | 小二号（18pt）黑体加粗，居中 |
| 一级标题 | 四号（14pt）黑体加粗 |
| 二级标题 | 小四号（12pt）黑体加粗 |
| 正文 | 小四号（12pt）宋体 |
| 首行缩进 | 2 字符（约 8.47mm） |
| 行距 | 1.5 倍（360 twips） |
| 页码 | 页脚居中，五号宋体 |

### 5.2 输出文件

| 文件 | 模式 | 说明 |
|------|------|------|
| `{名称}_技术交底书.docx` | 模式一 | 技术交底书 |
| `{名称}_请求书.docx` | 模式二 | 发明专利请求书 |
| `{名称}_权利要求书.docx` | 模式二 | 权利要求书 |
| `{名称}_说明书.docx` | 模式二 | 说明书（含附图说明） |
| `{名称}_说明书摘要.docx` | 模式二 | 说明书摘要 |

## 七、安装与部署

### 7.1 环境要求

- [OpenClaw](https://github.com/nicepkg/openclaw) >= 2026.3.2
- Node.js >= 18
- 已配置的 AI 模型（通过 OpenClaw 模型管理）

### 7.2 安装步骤

```bash
# 1. 进入 OpenClaw workspace 的 skills 目录
cd ~/.openclaw/workspace
mkdir -p skills && cd skills

# 2. 克隆项目
git clone https://github.com/CalicoCatto/PatentClaw.git patent-claw

# 3. 安装依赖
cd patent-claw && npm install
```

### 7.3 验证安装

通过 OpenClaw 对话发送包含触发词的消息（如"帮我写个专利"），如果龙虾进入 PatentClaw 模式则安装成功。

## 九、设计原则

### 9.1 专利思维优先

PatentClaw 始终以**最大化保护范围**为目标进行撰写，而非以论文的**完整公开复现**为目标。这体现在：

- 上位化改写：将具体实现抽象为功能性描述
- 多策略生成：提供不同保护范围的选择
- 独立权利要求精简化：只包含最核心的区别特征

### 9.2 绝不编造

系统绝不自行编造发明人、申请人、代理机构等身份信息。所有此类信息必须由用户提供，缺失时明确标注"待填写"。

### 9.3 初稿定位

PatentClaw 生成的所有文件均为初稿。每次输出完成后都会提醒用户：最终提交前须经持证专利代理师审核。
