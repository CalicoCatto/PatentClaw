# PatentClaw v2.0 — 多 Agent 编排架构

## 系统架构

PatentClaw 采用**编排器 + 4 个专业子 Agent** 的多 Agent 协作架构，基于 OpenClaw 的 `sessions_spawn` 原生子 Agent 能力实现。

```
                        ┌─────────────────────────┐
                        │     PatentClaw 编排器     │
                        │       (SKILL.md)         │
                        └────────┬────────────────┘
                                 │
              Phase 1: 交互式分析（编排器直接处理）
              ┌──────────────────┼──────────────────┐
              │                  │                  │
              ▼                  ▼                  ▼
        论文解析          创新点识别          上位化改写
              │                  │                  │
              └──────────────────┼──────────────────┘
                                 │
                        用户确认 + 收集申请人信息
                                 │
              Phase 2: 并行生成（3 个子 Agent 同时工作）
              ┌──────────────────┼──────────────────┐
              │                  │                  │
              ▼                  ▼                  ▼
    ┌─────────────────┐ ┌───────────────┐ ┌─────────────────┐
    │ Agent A          │ │ Agent B        │ │ Agent C          │
    │ 权利要求策略师    │ │ 说明书撰写师   │ │ 现有技术检索员    │
    │                  │ │                │ │                  │
    │ 输出：3套策略    │ │ 输出：说明书   │ │ 输出：查新报告    │
    │ (宽/稳/窄)      │ │ + 摘要         │ │ + 策略建议        │
    └────────┬────────┘ └───────┬───────┘ └────────┬─────────┘
              │                  │                  │
              └──────────────────┼──────────────────┘
                                 │
              Phase 3: 用户决策（编排器处理）
              ┌──────────────────┼──────────────────┐
              │                  │                  │
              ▼                  ▼                  ▼
        展示查新报告      展示3套策略对比     用户选择策略
              │                  │                  │
              └──────────────────┼──────────────────┘
                                 │
                          组装四件套 JSON
                                 │
              Phase 4: 自动审校（1 个子 Agent）
                                 │
                                 ▼
                    ┌─────────────────────────┐
                    │ Agent D                  │
                    │ 审校修复师                │
                    │                          │
                    │ 1. 运行一致性检查脚本     │
                    │ 2. 自动修复术语不一致     │
                    │ 3. 重新检查（最多3轮）    │
                    │ 4. 输出修正后的文档       │
                    └────────────┬─────────────┘
                                 │
              Phase 5: 导出交付（编排器处理）
                                 │
                                 ▼
                    ┌─────────────────────────┐
                    │ 展示最终文档 (Markdown)   │
                    │ 导出 .docx (国知局格式)   │
                    │ 检查报告摘要              │
                    │ 提醒用户找代理人审核       │
                    └─────────────────────────┘
```

## 子 Agent 详细说明

### Agent A: 权利要求策略师 (`claims-strategist`)

**职责**：生成 3 套不同保护策略的权利要求书

| 策略 | 独立权利要求特征数 | 从属数 | 保护范围 | 授权风险 |
|------|-------------------|--------|---------|---------|
| 宽保护 | 1-2 个 | 3-4 条 | 最大 | 较高 |
| 稳妥保护 | 3-4 个 | 4-6 条 | 适中 | 较低 |
| 窄保护 | 5-6 个 | 2-3 条 | 较小 | 最低 |

**为什么需要独立 Agent**：探索多种保护策略需要发散思维，与说明书撰写的收敛思维冲突。分离后各自在独立上下文中工作，生成质量更高。

### Agent B: 说明书撰写师 (`spec-writer`)

**职责**：撰写完整的说明书（技术领域、背景技术、发明内容、附图说明、具体实施方式）和说明书摘要

**为什么需要独立 Agent**：说明书需要大量的技术细节展开，占用大量上下文窗口。独立运行避免与权利要求生成竞争上下文资源。

### Agent C: 现有技术检索员 (`prior-art-scout`)

**职责**：使用 web_search 工具检索相似的专利和学术论文，输出查新报告和权利要求策略建议

**为什么需要独立 Agent**：网络搜索是 I/O 密集型操作，耗时较长。与其让主 Agent 等待搜索结果，不如让检索员在后台并行工作。其结果用于辅助用户在 3 套权利要求策略中做出更明智的选择。

### Agent D: 审校修复师 (`review-fixer`)

**职责**：运行 `check-consistency.js` 一致性检查脚本，自动修复发现的问题，最多迭代 3 轮直到 error 为 0

**为什么需要独立 Agent**：检查-修复-重检是机械性循环工作，不需要用户参与。交给独立 Agent 处理，编排器可以立即准备导出环境，减少用户等待时间。

## 通信协议

子 Agent 通过在输出中嵌入特定标记的 JSON 数据与编排器通信：

```
Agent A → __CLAIMS_RESULT__{...}__CLAIMS_RESULT__
Agent B → __SPEC_RESULT__{...}__SPEC_RESULT__
Agent C → __PRIOR_ART_RESULT__{...}__PRIOR_ART_RESULT__
Agent D → __FIXED_RESULT__{...}__FIXED_RESULT__
```

编排器从子 Agent 的输出文本中解析这些标记，提取结构化数据用于后续流程。

## 与单 Agent 方案的对比

| 维度 | 单 Agent（v1） | 多 Agent（v2） |
|------|---------------|---------------|
| 权利要求策略 | 生成 1 套 | 并行生成 3 套供选择 |
| 生成效率 | 串行（权利要求→说明书→检查） | 权利要求和说明书并行生成 |
| 查新能力 | 无 | 自动检索现有技术 |
| 术语审校 | 生成后手动检查 | 独立 Agent 自动检查+修复 |
| 上下文利用 | 所有工作共享一个上下文窗口 | 各 Agent 独立上下文，互不干扰 |
| 用户决策支持 | 无对比信息 | 查新报告 + 多策略对比 |

## 技术栈

- **运行时**：OpenClaw Agent Platform
- **子 Agent 调度**：`sessions_spawn` 原生工具
- **一致性检查**：`check-consistency.js`（Node.js）
- **文档导出**：`export-docx.js`（Node.js + docx 库）
- **网络搜索**：`web_search` 工具（Prior Art Scout 使用）
